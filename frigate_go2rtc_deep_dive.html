全局架构概览

Frigate 的三层处理架构与 go2rtc 的核心位置

Frigate 将视频监控系统拆分为三层：流接入层（go2rtc）、分析处理层（FFmpeg + Python AI）、输出层（WebRTC/RTSP/MQTT）。go2rtc 作为统一的视频网关，承担摄像头的多协议接入和流复用，让上游的 FFmpeg 和 AI 推理引擎只需消费标准 RTSP 流，无需关心摄像头原始协议。

摄像头管理

go2rtc 的流注册与 Frigate 摄像头角色分配机制

摄像头角色（Roles）系统

每个 Frigate 摄像头的 FFmpeg 输入可以被分配角色，决定该路流的用途。每个角色只能分配给一个输入流。

🔍
detect
目标检测主流，通常使用低分辨率子流，解码后送入帧采样 → 运动检测 → AI 推理管道
rtsp://127.0.0.1:8554/camera_sub
💾
record
录像流，使用高清主流，视频编码直接写入 MP4 文件，无需重新编码，最小化性能损耗
rtsp://127.0.0.1:8554/camera_main
🔊
audio
音频检测流（可选），用于声音事件检测（狗叫声、报警声等），音频流从同一 RTSP 中继拉取
rtsp://127.0.0.1:8554/camera_sub

两种典型摄像头接入模式

模式①：单流模式（节省摄像头连接数）
go2rtc.yaml + cameras
# 1. go2rtc 注册流（只连摄像头一次）
go2rtc:
  streams:
    hall_cam:
      - rtsp://admin:pass@192.168.1.5:554/live0
      - "ffmpeg:hall_cam#audio=opus"  # 转码音频

# 2. Frigate 摄像头从 go2rtc 中继拉流
cameras:
  hall_cam:
    ffmpeg:
      inputs:
        - path: rtsp://127.0.0.1:8554/hall_cam
          input_args: preset-rtsp-restream
          roles: [detect, record, audio]
📌
摄像头连接数：1 条（go2rtc 内部共享）
模式②：主子流模式（画质与性能平衡）
go2rtc.yaml + cameras
# 主流（高清）+ 子流（低清）分开注册
go2rtc:
  streams:
    hall_cam:          # 主流 2K
      - rtsp://admin:pass@192.168.1.5:554/main
    hall_cam_sub:      # 子流 720p
      - rtsp://admin:pass@192.168.1.5:554/sub

cameras:
  hall_cam:
    ffmpeg:
      inputs:
        - path: rtsp://127.0.0.1:8554/hall_cam
          roles: [record]     # 主流 → 录像
        - path: rtsp://127.0.0.1:8554/hall_cam_sub
          roles: [detect]    # 子流 → 检测
✅
推荐生产方案：主流高清录像保留画质证据，低清子流减少 AI 解码计算量，两者互不干扰。

go2rtc 与 Frigate 摄像头名称的自动映射

⭐
关键约定：当 go2rtc 中的流名称与 Frigate 摄像头名称完全一致时，Frigate 自动启用 MSE/WebRTC 高质量直播预览（优于 jsmpeg 低质量方案）。这是 Frigate 文档中推荐的最佳实践。

PTZ 摄像头支持（ONVIF）

Frigate 通过 ONVIF 协议管理 PTZ 摄像头（云台转动、变焦），go2rtc 处理视频流接入，ONVIF 控制是独立的 HTTP 控制通道。支持自动追踪（Autotracking）功能，可跟随运动目标自动转动摄像头。

go2rtc 在 Frigate 中的角色定位

go2rtc 做什么、不做什么，以及它与其他模块的边界

go2rtc 负责的事

能力说明
📡 多协议接入RTSP、RTMP、WebRTC、HomeKit、Tapo、小米、Ring 等 20+ 协议统一接入
🔀 流复用1 路摄像头连接分发给多个消费者（detect、record、audio、外部应用）
⚡ 低延迟输出WebRTC（:8555）、MSE 直接输出给浏览器，绕过 FFmpeg，延迟最低
🎙️ 双向音频中继通过 backchannel 支持 RTSP 双向音频（对讲），需独立流配置
🔄 按需转码仅在必要时（如 H265→H264）调用内置 FFmpeg 转码
📊 实时统计通过 :1984 Web UI 查看所有活跃流连接状态

go2rtc 不负责的事

不做由谁负责
❌ 帧解码与图像处理FFmpeg 进程负责（共享内存输出给 Python）
❌ 运动检测Python 进程（基于解码后的帧做帧差分）
❌ AI 目标检测独立检测器进程（TFLite/EdgeTPU/OpenVINO 等）
❌ 在视频流上叠加标注Frigate Python 后端（图像绘制 bbox）
❌ 录像文件写入FFmpeg record 进程（直写 MP4，避免重编码）
❌ 事件/告警发布MQTT 模块（Python 发布到 broker）
🔑
核心定位：go2rtc 是 Frigate 的「视频网关」，专注于协议接入和流复用，通过 RTSP 中继（:8554）向内部 FFmpeg 进程提供标准化视频源，同时通过 WebRTC/MSE（:8555/:1984）向外部客户端提供低延迟预览。AI 推理结果不流经 go2rtc，两者通过共享 RTSP 服务解耦。

视频处理流水线（Video Pipeline）

从摄像头到 AI 推理的完整 5 阶段处理步骤

1
视频采集（Feed Acquisition） go2rtc FFmpeg
go2rtc 作为第一道关口，将各种协议的摄像头统一转为标准 RTSP 流。FFmpeg 进程通过 rtsp://127.0.0.1:8554/<camera_name> 从本地 go2rtc 拉流，而非直连摄像头。每个摄像头只有一条原始连接，无论内部有多少消费者。

FFmpeg 根据 input_args: preset-rtsp-restream 预设以低延迟模式（无缓冲）拉取流，确保帧数据的实时性。
2
帧解码（Frame Decoding） FFmpeg 共享内存
FFmpeg 对 detect 角色的子流进行解码（H.264/H.265 → 原始 YUV/RGB 帧），可配置硬件加速（VAAPI/NVDEC/VideoToolbox）。

帧率控制：通过采样频率限制（如 5 FPS）降低后续处理压力，可在 detect.fps 配置。
分辨率控制：可配置 detect.width/height 对图像降采样。
输出方式：解码后的原始帧通过 共享内存（Shared Memory） 传递给 Python 检测进程，避免了进程间数据序列化开销。
3
运动检测（Motion Detection） Python 进程
Python 进程从共享内存读取解码帧，使用帧间差分算法（时序比较）检测画面变化。

检测流程：解码帧 → 像素差分 → 运动像素区域 → 合并为运动框（Motion Boxes）→ 聚合为运动区域（Motion Regions）

运动检测是 AI 推理的前置过滤器：只有存在运动的区域图像才会被裁剪并放入检测队列。静止画面不触发 AI 推理，大幅节省计算资源。可通过 MQTT 主题 frigate/<cam>/motion 实时获取运动状态。
4
AI 目标检测（Object Detection） 独立检测器进程
运动区域的裁剪图像被放入检测请求队列，多个摄像头共享同一检测队列，由独立的检测器进程处理。

支持的检测器后端：CPU TFLite（默认）、Google Coral EdgeTPU、OpenVINO（Intel GPU/CPU/NPU）、ONNX（Nvidia/AMD GPU）、Hailo-8 AI 加速卡。

检测器输出：边界框（bbox）+ 类别标签（label）+ 置信度分数（score）+ 属性

AI 增强功能（在检测结果基础上）：语义搜索、人脸识别、车牌识别（LPR）、鸟类分类、生成式 AI 描述（调用大模型对场景进行自然语言描述）。
5
快照与录像保存（Snapshot & Recording） FFmpeg record 进程
录像由专门的 FFmpeg record 进程处理：从 go2rtc 中继直接拉取原始编码的主流，写入 MP4 容器，不进行重新编码，最小化 CPU 开销。录制保留策略（按天/事件/对象类型）可独立配置。

快照由 Python 进程在原始帧上叠加检测标注（边界框、标签、置信度）后存储为 JPEG 文件。通过 HTTP API 端点 /api/<camera_name>/<label>/snapshot.jpg 可获取带标注的最新快照。

AI 推理详细流程

从帧图像到目标识别结果的完整链路

检测器进程架构

Frigate 的 AI 推理运行在独立的检测器进程中（与主 Python 进程分离），通过 Unix Domain Socket 或 共享内存 与主进程通信。多个摄像头的检测请求共享同一检测器队列，检测器串行处理后返回结果，避免了多模型实例的内存开销。

① 帧图像 → ② 运动区域裁剪 → ③ 图像预处理 （resize to 320×320 / 640×640） → ④ 模型推理 → ⑤ NMS 后处理 → ⑥ 结果写回主进程
NMS = Non-Maximum Suppression 非极大值抑制，去除重叠检测框

AI 推理后的视频输出

检测结果如何叠加回视频、以何种方式输出给用户

⚠️
重要架构认知：Frigate 的 go2rtc RTSP 重推流（:8554）不包含任何 AI 检测标注信息。go2rtc 输出的是摄像头原始流的透明转发，不会将边界框叠加到视频流中。检测标注只出现在：① Web UI 的实时预览画面、② 存储的快照 JPEG 文件。

三种视频输出路径详解

UI
Web UI 实时预览（带检测标注） jsmpeg MSE WebRTC
jsmpeg 低带宽流（默认 fallback）：Python 后端基于 detect 流解码帧，将 AI 检测结果（边界框、标签、置信度）直接绘制在原始帧上，编码为 MPEG1 格式通过 WebSocket 推送到浏览器。分辨率最高 720p，帧率最高 10fps，无音频。这是 Frigate Web UI 中显示检测标注的唯一实时方式。

MSE / WebRTC 高质量流（go2rtc 直出）：当 go2rtc 配置了对应流名称时，Frigate UI 优先使用 MSE 或 WebRTC。此时视频由 go2rtc 直接输出原始摄像头流到浏览器，不含 AI 标注，但分辨率/帧率/音频质量远优于 jsmpeg。Frigate UI 在此基础上叠加浏览器端 Canvas 绘制的检测框（通过 WebSocket 同步检测结果到前端）。

智能切换机制：无活动时显示静态帧 → 检测到运动后 1~2 秒内自动启动 MSE/WebRTC 高质量流 → 活动结束后销毁播放器，恢复静态帧节省带宽。
REC
录像文件（不含标注，原始画质） FFmpeg record 进程
FFmpeg record 进程从 go2rtc 拉取主流（高清原始编码流），直接写入 MP4 容器，不进行重新编码，不叠加任何标注。录制的是摄像头原始画质视频，配合 Frigate 事件系统可按事件分段查看。

录像文件存储在 /media/frigate/recordings/，可通过 Frigate Web UI 的 Timeline 视图浏览事件录像片段。录像保留策略（时长/触发条件）可独立配置。
SNAP
快照文件（含 AI 标注） Python 后端
Python 检测进程在原始帧上用 OpenCV 绘制边界框、类别标签、置信度数值，生成带 AI 标注的 JPEG 快照，每个追踪对象保存一张最高置信度时刻的快照。

访问方式：GET /api/<camera_name>/<label>/snapshot.jpg（实时最新）
历史快照：GET /api/events/<event_id>/snapshot.jpg
同步通过 MQTT frigate/<cam>/<label>/snapshot 以二进制形式发布。
EXT
外部应用消费（通过 RTSP 或 API） go2rtc :8554 HTTP API :5000
任何第三方应用（VLC、NVR、其他 AI 系统）可通过 go2rtc 的 RTSP 中继 rtsp://<frigate>:8554/<camera_name> 消费原始视频流。若需要叠加检测结果，外部应用需通过 HTTP API 或 MQTT 订阅检测事件，自行叠加。

双向音频（对讲）：通过 WebRTC（:8555）实现，需配置独立的 go2rtc 流并保留 backchannel：front_door_twoway: rtsp://...（不加 #backchannel=0）

输出路径对比

输出类型 含 AI 标注 协议/格式 延迟 音频 用途

生产可用的 Frigate + go2rtc 一体化配置

config.yml — 生产级多摄像头配置（主子流 + 对讲 + OpenVINO 检测器）
# ─── MQTT（数据交换出口）

  port: 1883
  topic_prefix: frigate

# ─── go2rtc（流接入网关）

  streams:
    # 前门摄像头（大华）：主流 + 子流分离
    front_door:
      - rtsp://admin:pass@192.168.1.100:554/main         # 主流 4MP
      - "ffmpeg:front_door#audio=opus"                  # 转码音频供 WebRTC
    front_door_sub:
      - rtsp://admin:pass@192.168.1.100:554/sub"          # 子流 720p
    front_door_twoway:
      - rtsp://admin:pass@192.168.1.100:554/main"         # 对讲专用（保留 backchannel）

    # 停车场摄像头（TP-Link Tapo，私有协议）
    parking:
      - tapo://admin:pass@192.168.1.101                   # Tapo 原生协议
    parking_sub:
      - "ffmpeg:tapo://admin:pass@192.168.1.101#video=h264&substream"

  # WebRTC 局域网候选地址
  webrtc:
    candidates:
      - 192.168.1.50:8555
      - stun:8555

# ─── AI 检测器（Intel GPU OpenVINO）
detectors:
  ov:
    type: openvino

架构要点总结

Frigate × go2rtc 完整集成的核心认知

七个关键认知
  • go2rtc 是视频入口，不是 AI 处理节点。它只做协议转换和流复用，将各品牌摄像头统一输出为标准 RTSP（:8554），不参与任何检测或标注，所有 AI 逻辑由 Python 进程独立处理。
  • 「1 摄像头连接，N 路消费」是 go2rtc 的核心价值。detect/record/audio 三个 FFmpeg 进程都从 rtsp://127.0.0.1:8554/camera_name 拉流，摄像头只需维持一条原始连接，大幅减少设备负担。
  • 运动检测是 AI 推理的前置门控。Python 进程基于共享内存帧做帧差分，仅当画面有运动时才裁剪图像送入 AI 检测队列，大幅减少 GPU/NPU 计算量。静止场景几乎不消耗推理资源。
  • 帧传递通过共享内存实现零拷贝。FFmpeg 解码后的原始帧直接写入共享内存，Python 检测进程直接读取，避免进程间管道传输的高带宽数据复制开销。
  • go2rtc 的 RTSP 中继不包含 AI 标注。外部系统通过 :8554 得到的是摄像头原始流，检测结果只通过 MQTT 发布或在快照/jsmpeg 流中体现。要在流媒体视频上叠加标注，需外部系统自行将 MQTT 检测数据与视频流结合渲染。
  • 录像不重编码是性能设计的关键决策。record 进程直接将摄像头原始 H.264/H.265 数据包写入 MP4 容器，完全不经过解码-再编码，保持摄像头原始画质且 CPU 开销接近零。
  • 双向音频需独立流实例避免冲突。go2rtc 连接 RTSP 流会占用 backchannel，影响对讲功能。需配置 front_door（加 #backchannel=0）和 front_door_twoway（默认不加）两个独立流，分别用于查看/录像和对讲。
